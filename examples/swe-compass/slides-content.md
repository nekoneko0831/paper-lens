## Slide 1: 标题页

**标题**：SWE-Compass：面向大语言模型智能体编程能力的统一评测
**副标题**：SWE-Compass: Towards Unified Evaluation of Agentic Coding Abilities for LLMs
**作者**：Jingxuan Xu, Ken Deng, Weihao Li 等
**机构**：快手科技 & 南京大学

**Speaker Notes**：大家好，今天分享一篇关于 LLM 编程能力评测的论文。我们都知道现在 AI 写代码越来越强了，但"到底强到什么程度"这个问题，现有的评测方法其实答得很不完整。这篇论文就是来解决这个问题的。

---

## Slide 2: 现有评测的困境

**核心信息**：现有评测只测了冰山一角

**内容**：
- SWE-bench 是目前最主流的编程评测：只测 Python + Bug 修复
- 真实开发中，程序员还要：写新功能、优化性能、写测试、做部署、代码重构...
- 而且不只用 Python——Java、Go、Rust、C++ 都是生产环境的主力

**视觉元素**：纯文字，冰山比喻结构

**Speaker Notes**：打个比方，现在的考试只考"用 Python 修 Bug"这一科。但真实工作中程序员要做八种事、用十种语言。只靠一科成绩来判断一个模型"编程能力强不强"，显然不够。

---

## Slide 3: 差距有多大？

**核心信息**：数据说话——现有基准的偏差触目惊心

**内容**：
- Python 占现有基准的 71.7%，其余 9 种语言合计不到 30%
- 6 种任务类型在开源基准中完全缺失
- 场景覆盖严重不均：Application Development 占 32.6%，UI/UX 仅 3%

布局：上文下图，重点看右半边的语言分布横条图

**Speaker Notes**：这是论文附录里的统计——他们标注了 5 个现有基准共 2400 多个实例。Python 独占七成，6 种任务类型完全空白。这不是小偏差，是系统性的盲区。

## Slide 4: SWE-Compass 全景

**核心信息**：一个覆盖全面的"全科试卷"

**内容**：
- 8 种任务：功能实现 / 功能增强 / Bug 修复 / 重构 / 性能优化 / 代码理解 / 测试生成 / 配置部署
- 8 种场景：应用开发 / 数据库 / 数据工程 / ML&AI / 基础设施 / 专业领域 / 安全 / UI/UX
- 10 种语言：Python, JS, TS, Java, C, C++, Go, Rust, Kotlin, C#
- 2000 个高质量实例，全部来自真实 GitHub PR

**视觉元素**：![Figure 2: 任务/场景/语言分布](figures/fig2-distributions.png)
布局：全图页，三个环形图并排

**Speaker Notes**：SWE-Compass 的回答是：全科考。8 种任务类型覆盖了开发者日常工作的方方面面，10 种编程语言对齐真实技术栈，每道题都来自 GitHub 上真实的代码合并请求。

## Slide 5: 四大设计原则

**核心信息**：不是随便凑 2000 道题，有严格的设计准则

**内容**：
- **真实性**：数据来自真实开发者对话和 GitHub PR
- **全面覆盖**：任务、场景、语言三个维度均衡分布
- **系统分类**：用 Active Learning 迭代发现分类体系
- **评测保真**：每个实例配可执行环境和测试

**视觉元素**：四个并列卡片，2×2 网格布局

**Speaker Notes**：这四个原则是 SWE-Compass 和其他基准的本质区别。特别是"评测保真"——每道题都有 Docker 环境，跑测试就能判对错，不存在主观评判的问题。

## Slide 6: 构建流水线总览

**核心信息**：五步流水线，从真实对话到最终数据集

**内容**：
- Step 1: 用户分析 → 定义"考什么"
- Step 2: 数据收集 → 从 GitHub 找"真题素材"（5万 PR）
- Step 3: 环境构建 → 配"考场"（4000 Docker 环境）
- Step 4: 任务构建 → 三种策略"出题"
- Step 5: 数据验证 → 质检 → 2000 实例

**视觉元素**：![Figure 3: 构建流水线](figures/fig3-construction-pipeline.png)
布局：全图页

**Speaker Notes**：整个构建过程是一条漏斗——从 5 万个 PR 开始，经过环境构建只剩 4000 个能跑通的，再经过任务构造和质检，最终只留下 2000 个。每一步都在筛掉不合格的。

## Slide 7: 怎么定义"考什么"

**核心信息**：用 Active Learning 从真实对话中自动发现分类体系

**内容**：
- 收集 Stack Overflow + GitHub 的开发者对话
- LLM + ICL 自动打标签
- 标签聚类 → 发现新类别 → 更新 → 再标注
- 迭代 5 轮收敛 → 8 任务 + 8 场景 + 10 语言

**视觉元素**：循环流程图，标注"5 轮迭代"

**Speaker Notes**：这个设计很聪明——不是人为规定"程序员的工作分这几类"，而是从几十万条真实对话里让数据自己告诉你。用主动学习迭代 5 轮，分类体系就自然收敛了。这保证了分类体系的客观性。

---

## Slide 8: 从 5 万 PR 到 4000 环境

**核心信息**：高质量筛选 + Docker 环境构建是最大的工程挑战

**内容**：
- 仓库筛选：500+ star、活跃维护、有测试、有许可
- PR 筛选：成功合并、有 Issue 描述、有明确改动
- 环境构建：自动提取依赖 → 生成 Dockerfile → 构建
- 残酷现实：自动成功率 2%，30 名专家修复后 8%

**视觉元素**：漏斗图，红色标注"2% → 8%"

**Speaker Notes**：这一步是整篇论文最"脏活累活"的部分。真实项目的依赖关系极其复杂，自动生成的 Docker 环境只有 2% 能跑通。请了 30 个专家花大量时间手动修复构建错误，才提到 8%。这也从侧面说明了真实软件工程的复杂度。

---

## Slide 9: 三种出题策略

**核心信息**：不同任务类型需要不同的构造方法

**内容**：
- **Checklist Synthesis** → 代码理解：LLM 出题 + 写评分检查点
- **Reverse Masking** → 配置部署/测试生成：故意弄坏正确答案
- **Heuristic Filtering** → Bug 修复/重构/新功能等：从真实 PR 按规则筛选

**视觉元素**：三列并排，每列一种策略 + 适用任务 + 原理

**Speaker Notes**：为什么需要三种策略？因为不同任务的"正确答案"获取方式不同。Bug 修复在真实 PR 里本来就有，直接筛就行。代码理解没有标准答案，需要 LLM 出题。配置部署的正确答案不好正向构造，所以反过来——先有正确的，故意弄坏，让模型修回来。

---

## Slide 10: 质检与最终数据集

**核心信息**：三道关卡确保每道题都靠谱

**内容**：
- 难度过滤：去掉太简单和太难的
- 均衡采样：按任务 × 场景 × 语言加权
- 人工验证：专家逐个确认可执行性和正确性
- 最终：2000 个高质量实例

**视觉元素**：纯文字

**Speaker Notes**：最后这一步是质检。先过滤极端难度，再做均衡采样保证分布合理，最后每道题都有专家验证。2000 个看起来不多，但每一个都是"出厂检验合格"的。

---

## Slide 11: 评测指标体系

**核心信息**：不同任务用不同指标，不搞一刀切

**内容**：

| 指标 | 适用任务 | 含义 |
|------|----------|------|
| Pass@1 | 功能实现/增强/Bug修复/重构 | 一次做对的概率 |
| 性能优化分数 | 性能优化 | 测试通过 + 时间降 20%+ |
| 行覆盖率 | 测试生成 | 测试能覆盖多少代码 |
| LLM-as-Judge | 代码理解 | 按 checklist 逐项打分 |

**视觉元素**：表格，简洁明了

**Speaker Notes**：一个关键设计——不是所有任务都用同一个指标。修 Bug 看"对不对"，优化性能看"快不快"，生成测试看"覆盖全不全"，理解代码看"答得准不准"。这种对齐让评测更贴合每种任务的实际目标。

## Slide 12: 主结果——谁考得最好？

**核心信息**：Claude-Sonnet-4 领先，任务难度分层清晰

**内容**：
- Claude-Sonnet-4 两个框架均第一（32.9% / 31.8%）
- 难度阶梯：代码理解 ~57% > 配置部署 ~51% > 功能增强 ~32% > Bug 修复 ~24% > 功能实现 ~17% > 性能优化 ~13%
- 多数模型 10-33%——基准确实很难

**视觉元素**：![Table 2: 主结果按任务类型](figures/tab2-main-results.png)
布局：全图页

**Speaker Notes**：先看总分——最强的也只有 33%，说明这个"全科考试"确实有难度。再看分科，代码理解最容易，性能优化最难。模型"读代码"还行，但"写新功能"和"让代码跑更快"就差很多。

## Slide 13: 两个框架谁更强？

**核心信息**：没有绝对赢家，各有所长

**内容**：
- SWE-Agent：edit-diff-execute 循环，擅长迭代定位型任务（Bug 修复）
- Claude Code：编辑器为核心 + 并行调用，擅长确定性任务（配置部署、测试生成）
- 交互轮次越多不一定越好，存在边际递减

**视觉元素**：左右对比布局，绿色标擅长任务

**Speaker Notes**：SWE-Agent 像侦探，擅长"反复排查、缩小范围"；Claude Code 像工匠，擅长"目标明确、一步到位"。而且轮次增多后收益递减，瓶颈不在于"多试几次"，而在于"第一次就找对方向"。

---

## Slide 14: 跨语言分析

**核心信息**：工具链成熟度比语言难度更影响模型表现

**内容**：
- 表现好：Java / Kotlin / JavaScript — 工具链成熟、反馈确定
- 表现差：C / C++ / Rust / Go — 构建复杂、错误信息不友好
- Python 意外中游 — 数据集刻意选了难题
- 强模型各语言都稳定（变异系数低）

**视觉元素**：![Figure 4: 各语言 Pass@1 对比](figures/fig4-passat1-by-language.png)
布局：全图页

**Speaker Notes**：Java 和 JavaScript 得分高不是因为语言简单，而是 JVM 和 Node 生态工具链成熟，构建稳定、错误提示清晰。C/C++/Rust 的构建系统更脆弱，模型经常卡在环境问题上。好的开发工具能放大模型能力。

---

## Slide 15: 失败模式——模型到底错在哪？

**核心信息**：瓶颈不在"不会写代码"，而在"没看懂要干什么"

**内容**：
- 对 3 个模型各 600 条失败轨迹归因
- 需求误解（RMI）：30-34%
- 方案不完整（ISE）：29-42%
- 两项合计 > 60%
- 技术知识欠缺（TKG）仅 5-8%

**视觉元素**：![Figure 7: 失败模式分布](figures/fig7-failure-modes.png)
布局：全图页，三个饼图并排

**Speaker Notes**：全文最有价值的分析。我们直觉以为模型做错题是因为"不会写代码"——但数据显示技术知识不足只占不到 8%。真正的问题是没看懂题目和改了 A 忘了 B。提升 Agent 的关键不在于教更多语法，而在于需求理解和全局设计能力。

## Slide 16: 附录精华① — Judge Prompt

**核心信息**：代码理解任务的"考官评分标准"

**内容**：
- 输入四要素：问题 + 评分 Checklist + 代码补丁 + 模型回答
- 评分规则：
  - 必须用自然语言解释，不能只贴代码 diff
  - 分数 = 满足的检查项 / 总检查项
  - 全部满足才给满分
- 扣分项：贴代码不解释、模糊表述、错误信息 → 0 分

**视觉元素**：左侧 Prompt 结构示意，右侧评分规则列表

**Speaker Notes**：附录 A.1 的 Judge Prompt。给考官四样东西，按清单逐条打勾算分。有个严格要求：只贴代码不解释直接 0 分。这个模板可以直接用于代码审查自动化。

## Slide 17: 附录精华② — 并行工具调用 Prompt

**核心信息**：一段简洁提示词让 Agent 效率大幅提升

**内容**：
- Claude Code SDK 附加的系统提示词
- 核心指令："独立操作时，同时发起所有工具调用"
- 示例：读 3 个文件 → 并行 3 个请求（1 倍耗时 vs 串行 3 倍耗时）
- 原则："宁可多并行，不要过度串行"

**视觉元素**：串行 vs 并行时间线对比图 + Prompt 原文节选

**Speaker Notes**：附录 A.2，只有一小段但效果显著。"能并行就并行"——同时读文件、同时跑命令。如果你在搭自己的 Agent 系统，这段 prompt 可以直接拿来用。

---

## Slide 18: 附录精华③ — 失败分析 Prompt

**核心信息**：让 LLM 当"事故调查员"的完整方案

**内容**：
- 关键设计：明确告诉 Judge"这条轨迹是失败的，忽略 Agent 的自我评价"
- 输入：问题描述 + 完整交互轨迹 + 工具列表
- 输出：描述 + 分类 + 出错动作
- 六个分类：需求误解 / 测试不充分 / 方案不完整 / 技术欠缺 / 工具错误 / 死循环

**视觉元素**：流程图 + 六分类色块

**Speaker Notes**：附录 A.3，最有实践价值的 Prompt。巧妙之处在于开头就强调"Agent 可能自以为成功了但它是错的"。模型经常"自我感觉良好"，这个 Prompt 绕开了这个陷阱。做 Agent 产品的话可以直接用来做错误归因。

## Slide 19: Key Takeaways

**内容**：
1. **评测要全面**：只测 Python 修 Bug 远远不够，软件工程是"全科考试"
2. **瓶颈在理解不在编码**：失败 60%+ 来自需求误解和方案不完整
3. **工具生态决定天花板**：语言工具链成熟度比语言本身难度更关键
4. **没有万能框架**：不同任务需要不同 Agent 工作流

**视觉元素**：4 个大号要点，简洁有力

**Speaker Notes**：四个 takeaway。第一评测要全面。第二最反直觉——瓶颈在理解需求不在写代码。第三好工具能放大模型能力。第四没有银弹，不同任务需要不同策略。

## Slide 20: 讨论与未来方向

**内容**：
- 局限：Docker 成功率 8%、部分任务用了 LLM 合成
- 未来：更多语言、跨模块长上下文任务、人工校准 LLM-as-Judge
**视觉元素**：上半局限性，下半未来方向

**Speaker Notes**：8% 的环境构建成功率是最大工程瓶颈。留一个讨论问题：基于刚才的失败模式分析，你们觉得自己在用的编程 Agent 最可能卡在哪种错误上？
