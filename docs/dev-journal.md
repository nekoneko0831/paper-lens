# Paper Lens 开发日志

> 从评测 7 个现有 Skill 到自己造一个的全过程。

## 起源

事情的开头很简单：我想用 Claude Code 读论文。不是那种"帮我翻译一下摘要"的读法，而是真正的三步走——快速判断这篇论文值不值得读、深入理解它到底在说什么、准备一场能给团队讲明白的分享。

市面上已经有不少论文阅读相关的 Claude Code Skill。我前后测试了 7 个：ai-paper-reader、deep-reading-analyst、scientific-slides、paper-summarizer，还有几个散落在各种 awesome 列表里的。每一个都有自己的亮点——有的速览做得很好，有的深度分析很到位，有的 slides 生成很专业。但没有一个能满足"快速消化 + 深度理解 + 准备汇报"的完整链路。

我的需求说出来其实很朴素：读完一篇论文后，我既要能在 5 分钟内判断它值不值得深读，又要能花半小时真正搞懂它（不是翻译，是用大白话讲明白），最后还要能直接拿着输出准备一场 20 分钟的团队分享。三步之间还要能衔接——速览的内容不用在学习模式里重复生成。

现有 Skill 做不到这一点，于是决定自己动手。

## 核心设计决策

Paper Lens 的设计过程中，有几个关键的分叉口：

**三模式架构 vs 单模式全输出。** 最初考虑过一次性输出所有内容（速览+深度+slides），但很快发现这行不通——单次输出太长 Claude 会截断，而且用户大部分时候只需要速览。三模式拆分后，每个模式的输出可以控制在合理长度，模式之间还能串联复用。

**"大白话"风格 vs 学术翻译风格。** 学习模式的定位从一开始就很明确：不是翻译论文，是"用自己的话讲明白"。这意味着不能逐段翻译原文——要先给直觉，再给细节，用类比来解释抽象概念。实际效果差异很大：翻译风格读起来和读原文一样累，大白话风格读完真的能理解。

**增量保存 vs 一次性输出。** 学习模式是多轮交互的，每一步（术语解释、方法拆解、公式解读）都可能花好几分钟。如果最后才一次性保存，中途断开就全丢了。增量保存（每步完成自动追加到文件）解决了这个问题，用户全程可以打开文件查看已学内容。

**AskUserQuestion 分组多选 vs 多步确认。** 术语和公式的选择环节，最初设计的是"先问要不要自己选，再出选项"——两步确认太啰嗦。改成直接出分组选项后，一步到位，体验好很多。不过 AskUserQuestion 有个限制：每个问题最多 4 个选项。所以需要按相关性分组，每组不超过 4 个。

**双层公式输出。** 终端里显示 LaTeX 公式可读性很差，但保存到文件里用 ASCII 又不够精确。最终方案是两层：终端用 ASCII 渲染方便实时阅读，文件里保存 LaTeX 版本方便后续引用。

**名词分两层。** 论文里的术语分两类：论文专有的（如"SWE-Compass"、"Checklist Synthesis"）和领域通用的（如"Pass@1"、"Active Learning"）。前者需要结合论文上下文解释，后者可以给更通用的定义。分层后读者可以快速区分哪些是"这篇论文发明的"、哪些是"这个领域通用的"。

## 迭代过程

**v0.1：速览模式。** 第一个版本只有速览，用 SWE-Compass 论文测试。基本信息卡片 + TL;DR + 核心方法 + 实验结果。输出质量不错，但发现缺少深度问答环节——加上 5 个常见问题后，速览的价值感明显提升。

**v0.2：学习模式。** 加入增量保存和术语交互选择。第一轮测试暴露了几个问题：术语解释太短（一句话不够）、方法拆解缺少整体流程图、没有公式环节。逐一修复后，学习模式的笔记从"读完就忘"变成了"可以回头翻看的参考资料"。

**v0.3：展示模式。** 最具挑战性的模式。要考虑听众背景、时间分配、每页的核心信息和 Speaker Notes。最大的痛点是图表——slides 里必须引用论文原图，但提取出来的图片文件名是 `p3_fig1.png` 这种格式，完全看不出内容。于是设计了图表映射流程：识别 → 重命名（`fig1-architecture.png`）→ 复制到 `figures/`。

**v0.4：图片提取脚本升级。** 最初的 `extract_figures.py` 只做基础提取，遇到跨列图（一张图分成两半提取）和嵌入位图（低分辨率小图）时效果很差。升级后加入了三个关键改进：矩形合并（merge_rects，把拆开的图拼回去）、Caption 检测（根据图片下方文字判断这是 Figure 几）、全页宽扩展（检测到 Caption 后把裁剪范围扩展到全页宽，解决跨列问题）。同时输出 `manifest.json`，记录每张图的来源页码、尺寸和类型，方便程序化访问。

**v0.5：HTML 演示文稿集成。** slides-content.md 只是内容文档，真正要做分享还需要一个 HTML 版本。集成 frontend-slides 后，关键问题是图片路径——HTML 里用相对路径引用图片，发给别人就看不到了。最终方案是 base64 编码嵌入，HTML 文件完全自包含，发给谁都能打开。

**v0.6：第二轮测试。** 用 tau2-bench 论文做端到端测试。暴露了一个学习模式的问题：讨论页（Discussion）的输出有模板感，每篇论文的讨论部分都长得差不多。修改了 prompt，让讨论基于论文的具体贡献和局限性展开，而非套用通用框架。展示模式也做了调整——发现 Speaker Notes 太长影响排版，精简到每页 3-5 句。

## 测试经验

两篇论文的端到端测试暴露了不少问题：

SWE-Compass 是一篇评测论文，结构清晰、图表丰富，是很好的第一个测试对象。最大的收获是发现了图表提取的各种边界情况——跨列图、嵌入在文字里的小图标、表格截图。速览 → 学习 → 展示的全链路走通后，输出质量基本达标。

tau2-bench 是一篇方法论文，公式更多、结构更复杂。学习模式在处理多个核心公式时遇到了排版问题（ASCII 公式太长导致终端换行混乱），通过缩短 ASCII 表示和调整对齐方式解决。展示模式在这篇论文上效果更好——方法论文天然适合做技术分享。

## 开源准备

把一个"自用 Skill"变成"其他人也能用的 Skill"，主要做了三件事：

**解耦 frontend-slides 依赖。** Paper Lens 的核心功能（三模式读论文）不依赖 frontend-slides。HTML 演示文稿是可选的增强功能，在 README 里标注为 optional dependency。

**修复脚本路径可移植性。** `extract_figures.py` 最初硬编码了项目路径。改成接受命令行参数后，安装到任意目录都能正常工作。SKILL.md 中的路径引用也做了相应调整。

**添加双语文档。** Skill 本身的输出全是中文，但开源后需要让英文用户也能理解它在做什么。README 采用英文优先、中文详细的双语结构。中文部分不是翻译，而是面向中文用户重新组织的说明，包含更多使用细节和触发方式。
